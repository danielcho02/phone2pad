# 00 — OVERVIEW: 목표와 성공 기준

## 1. 문제 정의

노트북에는 Windows Precision Touchpad(WPT)가 있어 3/4손가락 제스처가 OS 네이티브로
동작하지만, 데스크탑에는 그런 입력 장치가 없다. 기존 "폰을 마우스로" 앱들
(Mousedroid, Monect, Remote Mouse 등)은 전부 **마우스 에뮬레이션** 수준이며,
OS 제스처(Task View, 가상 데스크탑 전환)를 지원하지 않는다.

## 2. 목표 (Goals)

| # | 목표 | 측정 방법 |
|---|---|---|
| G1 | 폰을 USB-C로 연결하면 Windows가 **Precision Touchpad로 인식** | 설정 > Bluetooth 및 장치 > 터치패드 페이지에 제스처 설정 UI가 나타남 |
| G2 | 1손가락 이동/탭, 2손가락 스크롤/우클릭, 핀치 줌 | 수동 테스트 시나리오 통과 |
| G3 | 3손가락 위로 = Task View, 4손가락 좌우 = 가상 데스크탑 전환 | Windows 기본 제스처 매핑 그대로 동작 |
| G4 | 폰 화면은 **완전 검정, UI 없음** | 트랙패드 사용 중 화면에 픽셀 출력 없음 |
| G5 | 입력 지연 체감 불가 수준 | 터치 → 커서 반응 p95 ≤ 20ms (측정 도구 포함) |
| G6 | 폰 좌표계 전체를 패드 표면으로 사용 (절대 → 상대 변환은 OS/드라이버 표준 방식) | 폰 어느 위치에서 시작해도 트랙패드처럼 동작 |

## 3. 비목표 (Non-Goals)

- Wi-Fi / Bluetooth 연결 (USB 전용)
- macOS / Linux 호스트 지원 (v1은 Windows 10/11 전용)
- 폰 쪽 제스처 커스터마이징 UI
- 키보드 / 게임패드 기능
- Play Store 배포 품질의 폰 앱 (개인 사이드로드 전제)

## 4. 사용자 시나리오 (수락 시나리오)

1. 폰을 USB-C로 PC에 연결한다. (USB 디버깅 사전 허용 상태)
2. PC 서비스가 자동으로 폰 앱을 기동시키고 ADB forward를 설정한다.
3. 폰 화면이 꺼진 것처럼 검게 변한다. 이때부터 폰 = 트랙패드.
4. 두 손가락으로 웹페이지를 스크롤하고, 세 손가락을 위로 쓸어 Task View를 열고,
   네 손가락으로 옆 가상 데스크탑으로 전환한다.
5. 케이블을 뽑으면 폰은 평소 상태로 돌아오고 PC의 가상 트랙패드 장치는 제거된다.

## 5. 제약 조건

- **드라이버 서명**: 자체 KMDF 드라이버는 테스트 서명 모드 필요.
  개인 사용 전제이므로 허용. (배포 시 EV 서명 비용은 범위 외)
- **PTPHQA 인증 블롭**: Windows는 PTP 장치에 인증 feature report를 요구한다.
  공개된 테스트 블롭 사용 (상세는 `03-HID-PTP-SPEC.md`).
- **Android 권한**: raw 터치는 일반 Activity의 `MotionEvent`로 충분 (특수 권한 불필요).
  앱이 포그라운드여야 하므로 검정 전체화면 Activity + 화면 밝기 0 전략 사용.

## 6. 참고 선행 프로젝트

| 프로젝트 | 가져올 것 | 한계 |
|---|---|---|
| `darusc/Mousedroid` | ADB-USB 연결 구조, Kotlin 터치 캡처 패턴 | 마우스 수준, OS 제스처 없음 |
| `imbushuo/mac-precision-touchpad` | **PTP HID 디스크립터 + 드라이버 구조의 최고 레퍼런스** (Magic Trackpad를 WPT로 인식시키는 Windows 드라이버) | 입력원이 실제 USB/BT 하드웨어 |
| `djpnewton/vmulti` | 가상 HID 멀티터치 드라이버의 고전 레퍼런스 | 오래됨, PTP 아님 (digitizer) |

> 에이전트 주의: 위 레포들의 코드를 그대로 복사하지 말 것 (라이선스 확인 필요).
> 구조와 디스크립터 필드 구성을 학습 레퍼런스로만 사용하라.
